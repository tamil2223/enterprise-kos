## RAG Playground UI + API — Design Spec (v1)

### Goal
Build an interactive UI that demonstrates **how a RAG pipeline works** end-to-end:
ingestion → chunking → embedding → indexing → retrieval → (prompt assembly).

This feature is **educational + debuggable**, not production retrieval quality.
It should make it easy to compare “good vs bad” chunking choices (especially **overlap vs no-overlap**).

### Non-goals (v1)
- PDF ingestion (defer)
- External embedding providers (Gemini/OpenAI) and local embedding models (defer; keep interface ready)
- Cross-encoder reranking (defer; leave a rerank slot)
- Persistent storage (runs stored in memory; reset on restart)
- Auth/RBAC (defer)

### Future planned upgrades (v2+)
These are explicitly out of v1, but the v1 interfaces should make them easy to plug in.

#### Chunking upgrades
- **Token-based chunking**: choose chunk sizes in tokens (not chars) using a tokenizer aligned to the embedding/LLM model.
- **Structure-aware splitting**:
  - Markdown-aware splitters (headers, lists)
  - Code-aware splitters (preserve blocks)
  - “Semantic-ish” splitters (sentence boundaries / section boundaries)
- **Better overlap strategy**:
  - overlap by tokens
  - overlap only around boundary regions (avoid duplicating large repeated regions)
- **Metadata enrichment**:
  - section titles / heading path
  - source URL / document version
  - stable chunk IDs across re-chunk runs when possible

#### Embedding upgrades (real vectors)
- **Local embeddings** (no keys): `sentence-transformers` 384-dim model (e.g., MiniLM family)
  - same `dim=384` contract; replace fake embedder implementation
- **Provider embeddings** (production-like):
  - **Gemini embeddings** (requires key)
  - **OpenAI embeddings** (requires key)
- **UI enhancements for real embeddings**:
  - show vector norms + cosine similarity distribution
  - optional 2D projection (PCA/UMAP) for intuition (kept off by default for performance)

#### Retrieval quality upgrades
- **Hybrid retrieval**: sparse BM25 + dense embeddings, fusion with configurable weights
- **Reranking**: cross-encoder scoring on the top-N retrieved chunks before final top-k
- **Evaluation loop**: small labeled query set + metrics (hit@k, MRR, nDCG) to compare chunking/embedding choices

## Architecture

### Components
- **FastAPI backend (existing)**: add `/playground/*` routes that execute each pipeline step and store per-run artifacts in memory.
- **Next.js frontend (existing `web/`)**: a “stepper” UI that calls `/playground/*` endpoints and visualizes artifacts.

### Run-based state model
The playground is organized around a **run**:
- A user creates a `run_id`
- Each step mutates state for that `run_id` (documents, chunks, embeddings, index)

This makes the workflow debuggable and avoids huge client payloads.

## Data model (backend, in-memory)

### RunState
- `run_id: str`
- `created_at: float`
- `documents: dict[doc_id, Document]`
- `chunks: dict[doc_id, list[Chunk]]` (generated by chunking)
- `embeddings: dict[doc_id, EmbeddingState]` (generated by embedding)
- `index: VectorIndexState | None` (generated by index step)

### Document
- `doc_id: str`
- `name: str`
- `content_type: "text/plain" | "text/markdown"`
- `text: str`

### Chunk
- `chunk_id: str`
- `doc_id: str`
- `chunk_index: int`
- `start_char: int`
- `end_char: int`
- `text: str`

### EmbeddingState
- `dim: 384`
- `vector_by_chunk_id: dict[chunk_id, list[float]]` (length=384 each)
- `vector_id_by_chunk_id: dict[chunk_id, str]`

### VectorIndexState
- `dim: 384`
- `vector_ids: list[str]`
- `vectors: list[list[float]]`
- `chunk_id_by_vector_id: dict[str, str]`
- `doc_id_by_vector_id: dict[str, str]`

## Chunking design (v1)

### Why change from current `chunk_paragraphs`
The existing backend chunker is paragraph-pack based and does not support overlap.
For the playground, we need a chunker that matches the UI controls:
- **chunk_size**
- **overlap**

### Chunker behavior (window + overlap)
- Inputs:
  - `text: str`
  - `chunk_size_chars: int` (e.g. 400–2000)
  - `overlap_chars: int` (e.g. 0–400, must be `< chunk_size_chars`)
- Algorithm:
  - Normalize whitespace minimally (preserve readability; collapse repeated spaces, standardize newlines)
  - Create chunks by sliding window:
    - first chunk: `[0 : chunk_size]`
    - next chunk starts at: `prev_end - overlap`
  - For readability, adjust the chunk end to the nearest newline boundary if found within a small window (optional in v1).
- Output:
  - deterministic `chunk_id = f\"{doc_id}#chunk-{i}\"`
  - store `start_char`, `end_char`

### Comparison feature (core UX)
UI will allow toggling:
- **overlap=0** vs **overlap>0**
and demonstrate retrieval differences for the same query (top-k chunk set changes).

## Embeddings design (v1)

### Fake embeddings (deterministic, 384-d)
We use deterministic fake embeddings so:
- no keys required
- stable demos
- easy to validate visually

Requirements:
- Same chunk text always maps to the same 384-d vector
- Similar texts should be “somewhat” similar (best-effort) but not perfect

Implementation sketch:
- Tokenize chunk text to words
- Hash tokens into a 384-d vector bucket space
- Use signed hashing and L2 normalization

Later swap:
- Local `sentence-transformers` 384-d
- Gemini embeddings

## Index + retrieval (v1)

### Index
Build an in-memory vector store per run:
- store all vectors + mapping `vector_id ↔ chunk_id`

### Retrieval
- Embed the query (same fake embedder)
- Similarity:
  - cosine similarity (dot product after normalization)
- Return:
  - top_k (default 5)
  - `score`, `vector_id`, `chunk_id`, `doc_id`, `chunk_preview`

### (Deferred) Rerank slot
Design should allow adding:
- `rerank_candidates(query, chunks)` in the future
without breaking API responses.

## Backend API (v1)

### Runs
- `POST /playground/runs`
  - Response: `{ "run_id": "..." }`
- `GET /playground/runs/{run_id}`
  - Response: counts + completed steps

### Documents
- `POST /playground/runs/{run_id}/documents:load-sample`
  - Loads 2–5 built-in sample docs
- `POST /playground/runs/{run_id}/documents:upload`
  - Multipart upload; accept `.txt` and `.md` only
- `GET /playground/runs/{run_id}/documents`
  - Returns document list

### Chunking
- `POST /playground/runs/{run_id}/chunk`
  - Body: `{ "doc_id": "...", "chunk_size": 800, "overlap": 120 }`
  - Response: chunk stats + preview list
- `GET /playground/runs/{run_id}/chunks?doc_id=...`
  - Response: chunks

### Embeddings
- `POST /playground/runs/{run_id}/embed`
  - Body: `{ "doc_id": "..." }`
  - Response: `{ dim, mapping_preview, vectors_preview }`

### Index
- `POST /playground/runs/{run_id}/index`
  - Response: `{ indexed, mapping_preview }`

### Retrieval
- `POST /playground/runs/{run_id}/retrieve`
  - Body: `{ "query": "...", "top_k": 5 }`
  - Response: `{ query_vector_preview, results:[...] }`

## Frontend UX (v1)

### Single-page stepper
Steps shown in a left sidebar:
1. Ingestion
2. Chunking
3. Embeddings
4. Index
5. Retrieval

Right pane shows:
- inputs for step
- artifacts (tables, previews, highlighting)

### Artifact visualization requirements
- Chunks: list with `chunk_id`, offsets, preview; click chunk to view full text
- Embedding matrix: show a preview (e.g., first 10 vectors × first 16 dims) + vector norms
- Index mapping: show chunk_id → vector_id mapping preview
- Retrieval: show top-k results and highlight the matched chunks
- Comparison toggle: “overlap 0” vs “overlap X” retrieval results side-by-side for same query

## Testing (v1)
- Backend unit tests:
  - chunking produces expected number of chunks and correct overlap boundaries
  - fake embedder returns 384-d vectors and deterministic output
  - retrieval returns stable top-k shape and non-empty results on sample docs
- Basic API tests:
  - create run → load sample → chunk → embed → index → retrieve (happy path)

---

## Success criteria
- UI can run the full pipeline on sample docs without manual setup
- Chunking overlap toggle visibly changes the retrieved chunks for some queries
- No external keys required (fake embeddings)
- Backend remains responsive with moderate doc sizes (playground scope)
